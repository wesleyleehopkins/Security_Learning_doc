FIELD TYPE DESCRIPTION · WHY IT MATTERS · LINKS
@timestamp date ISO8601 timestamp of when this event was indexed into Elasticsearch.
Use for time-bounding searches. Note: this may differ slightly from the actual event time — check @timestamp vs event time in the raw message for imported logs.
🔗 Common to all datasets
@version keyword Logstash/Beats pipeline version string.
Infrastructure metadata — not useful for analysis. Usually "1".
🔗 Common to all datasets
client.ip ip IP address of the connection originator (client side).
Same as source.ip in most cases. Use for filtering inbound/outbound connections from the client perspective.
🔗 Cross-ref: source.ip, zeek.http, zeek.dns
client.ip_bytes long Total IP-level bytes sent by the originator including headers.
High value = large upload or sustained data send. Low = minimal client activity (check-in/beacon). Compare with server.ip_bytes for direction.
🔗 Compare: server.ip_bytes
client.oui keyword Hardware vendor from MAC OUI of the originating device.
Passive asset fingerprinting. Unexpected vendors (NETGEAR on servers) = anomaly. Only visible if sensor is on same L2 segment.
🔗 Asset inventory, anomaly detection
client.packets long Number of packets sent by the originator.
1 packet = probe/scan attempt. Regular packet counts at intervals = beaconing pattern.
🔗 Beaconing detection: group by dst + count over time
client.port integer Ephemeral source port used by the originator.
High ports (1024-65535) are normal for clients. Very low source ports are unusual and could indicate spoofing.
🔗 Context: destination.port is more significant for service identification
connection.bytes.missed long Bytes Zeek could not capture — gaps in packet capture.
Non-zero means the sensor missed traffic. File hashes in linked zeek.file will be unreliable. PCAP will be incomplete.
🔗 Affects: zeek.file hash reliability, PCAP completeness
connection.history keyword Letter-coded sequence of TCP events seen (S=SYN, h=SYN-ACK, A=ACK, D=data, R=RST, F=FIN).
Read left to right — tells you the full lifecycle of the connection. "R" alone = RST without SYN (midstream or scan reject). "ShAdDaFf" = clean bidirectional session.
🔗 Explains: connection.state value in plain sequence form
connection.local.originator boolean True if the connection originated from inside the monitored network.
false + local.responder=true = inbound from external — always check destination.port. Combine both fields to determine full connection direction.
🔗 Pair with: connection.local.responder
connection.local.responder boolean True if the connection destination is inside the monitored network.
true = external host connecting TO your internal systems. Critical for identifying inbound C2 callbacks, exploitation, or scanning.
🔗 Pair with: connection.local.originator
connection.state keyword Zeek connection state code (SF, REJ, S0, OTH, RSTO, etc.).
SF=clean session, REJ=immediately rejected, S0=SYN no response, OTH=midstream/no SYN. REJ+S0 in volume = port scan.
🔗 Read alongside: connection.state_description for plain English
connection.state_description keyword Human-readable explanation of the connection.state code.
Plain English version — much faster than memorizing state codes. Always shows exactly what Zeek observed in the flow lifecycle.
🔗 Directly explains: connection.state
container.id keyword Internal SO container identifier for the log pipeline.
Infrastructure metadata — not useful for threat analysis.
